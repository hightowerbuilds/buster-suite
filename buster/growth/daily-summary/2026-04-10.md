### Pretext.js Discovery

`@chenglou/pretext` is declared in `buster/package.json` but **never imported anywhere in the codebase**. Text measurement is handled by `OffscreenCanvas.measureText()` in `src/editor/text-measure.ts`. Both approaches are lightweight and avoid DOM layout, but the README claims Pretext.js specifically. This is a documentation/marketing mismatch, not a minimalism failure.

### What's Working Well (Minimalism)

- **Canvas-only rendering** — Editor, terminal, welcome screen, tour, and even the marketing website all render to Canvas 2D. Zero DOM text nodes for content.
- **SolidJS signals** — Used purely for state management and triggering canvas redraws. No component libraries, no virtual DOM.
- **Zero-dep supporting crates** — buster-path, buster-sandbox, buster-test-harness, buster-lsp-manager, buster-syntax, buster-terminal-pro, buster-remote, buster-dap all have zero or near-zero production dependencies.
- **No framework bloat** — No Tailwind, Bootstrap, Material Design, styled-components, lodash, moment, jQuery, or icon libraries.
- **11 MB installer** — Exceptional for a full-featured IDE with canvas rendering, terminal, git, AI, LSP, extensions, and syntax highlighting.

### Areas of Concern (Quality, Not Minimalism)

These don't violate minimalism but affect production readiness:

1. **DAP client UB** — `debugger/client.rs:42-46` has raw pointer cast with unsafe Send/Sync
2. **Tauri FS scope `**`** — Frontend can read/write any file on system (`capabilities/default.json`)
3. **AI command blocklist** — Substring matching in `ai/tools.rs` is trivially bypassable
4. **CRDT transform bug** — `collab/crdt.rs` Insert vs Delete returns zero instead of shift
5. **No incremental parsing** — Full tree-sitter reparse on every keystroke
6. **Full-document LSP sync** — Entire file sent on every change
7. **No `didClose` notifications** — LSP servers leak memory over time

## Session 2 — Production Build, Extension System Fixes, Manifest-Driven UI

### Production Build

Built Buster IDE for macOS production release. Generated `Buster_0.1.0_x64.dmg` (14 MB) and placed it on the Desktop for friend testing.

### Extension System Bug Fixes

**Uninstall not working:** `ext_uninstall` and `ext_install` used `app.path().app_config_dir()` to compute the extensions directory, which resolved to a different path than `~/.buster/extensions/` used by the scanner. Fixed both to use the shared `extensions_dir()` helper. Also fixed uninstall to clean up `state.json` so uninstalled extensions don't auto-restore on next launch.

**WASM notify signature mismatch:** The host linked `notify` as `(i32, i32, i32, i32) -> i32` but the Ext-rust extensions declared it with no return value. Wasmtime is strict about type matching, causing instantiation failures. Fixed the host to return void.

**Epoch deadline expiry:** The 5-second WASM execution timeout was set once at instantiation and never reset. Any command called more than 5 seconds after loading would immediately trap. Fixed by resetting the epoch deadline before each `call_method`.

**Browser export name mismatch:** The browser extension exported `render_page_cmd` but the manifest command ID was `render_page`. Renamed the export to match.

### OCaml Extensions Removed

Removed `buster/extensions/extOCaml/` directory from the monorepo. Extensions are now built in Rust via the separate Ext-rust repo.

### Manifest-Driven Extension UI (kind field)

Added a `kind` field to `[[commands]]` in `extension.toml`. Values:
- `"launch"` — renders as a button on the extension card that auto-enables and opens a tab
- `"palette"` (default) — only appears in the command palette

Changes across both repos:

**Buster (IDE):**
- `manifest.rs`: `CommandDecl` now has `kind` field (defaults to `"palette"`)
- `mod.rs`: `ExtensionCommand` includes `kind`, passed through from manifest
- `ipc.ts`: Frontend `ExtensionCommand` type includes `kind`
- `ExtensionsPage.tsx`: Extensions with `kind = "launch"` commands show Launch buttons instead of Enable/Disable toggle. Clicking Launch auto-enables if needed, then calls the command.
- `extensions.css`: Added `.ext-launch-btn` styles

**Ext-rust (extensions):**

- Browser internal `render_page()` renamed to `build_page()` to avoid conflict with the export

## Session 2b — Allocator Fix, Surface Paint Race Condition

### WASM Allocator Stack Overflow (Critical)

**Problem:** Clicking Launch caused a WASM trap with ~8000 recursive frames of `wasm function 7` followed by "call stack exhausted."

**Root Cause:** `SimpleAllocator` (the `GlobalAlloc` implementation in `shared/src/runtime.rs`) used `Vec::with_capacity()` inside its `alloc()` method. Since `Vec` uses the global allocator, this created infinite recursion: `GlobalAlloc::alloc` → `Vec::with_capacity` → `GlobalAlloc::alloc` → ...

**Fix:** Replaced with a bump allocator that uses WASM's `memory.grow` intrinsic directly:
- Tracks heap position via `AtomicUsize` static
- Lazily initializes from the `__heap_base` WASM global
- Grows WASM linear memory in 64KB pages as needed
- No-op dealloc (memory freed when WASM instance is dropped)

The `alloc` export function was also updated to go through the `GlobalAlloc` trait instead of using `Vec`.

### Surface Paint Race Condition

**Problem:** Extensions launch into tabs but browser and stacker show empty pages. Pixel editor renders because it happens to have simpler timing.

**Root Cause:** When the WASM extension calls `host_request_surface` followed by `host_paint`, both happen synchronously within the same `call_method` invocation. The "created" event tells the frontend to open a tab, but the "paint" event arrives before the `DisplayListSurface` component has mounted and set up its event listener. The paint data is lost.

**Additional issue:** Each `ext_load` call created a **new** `SurfaceManager` instance. The `surface_get_last_paint` command queried the global managed instance, which was a different object with no data.

**Fix (Buster side):**
1. `SurfaceManager` now buffers the last paint content per surface in `last_paint: Option<String>`
2. Added `get_last_paint(surface_id)` method and `surface_get_last_paint` Tauri command
3. `DisplayListSurface` calls `surfaceGetLastPaint` on mount to fetch any buffered content
4. `ExtensionManager` now owns a single shared `SurfaceManager` via `shared_surface_manager: Arc<SurfaceManager>`
5. `ext_load` and `ext_restore` use `state.surface_manager()` instead of creating new instances
6. All extensions share the same surface registry, so buffered paints are always findable

## Canvas Color Saturation Bug Fix

### Problem
Scrolling in the editor caused colors to gradually diminish and saturate — the image degraded with each frame of scroll input.

### Root Cause (Two Layers)

**1. Canvas context state leaking between frames.** The 2D context carries persistent state (`globalAlpha`, `shadowBlur`, `shadowColor`, `globalCompositeOperation`, `filter`). The cursor glow effect sets `shadowBlur` and `shadowColor` via `applyCursorGlow()`, and while `clearCursorGlow()` resets these after drawing cursors, any early return or exception path could skip the cleanup. Over hundreds of scroll frames, leaked shadow state compounded and visually degraded the rendered image.

**2. Wheel event flooding.** A single trackpad swipe fires 20-60 `wheel` events per frame. Each event immediately called `setScrollTop()`, triggering the SolidJS reactive effect and `scheduleRender()`. While the rAF guard prevented multiple renders per frame, the repeated signal writes and effect re-evaluations added unnecessary overhead and timing pressure.

### Fix

**Context state reset** (`canvas-renderer.ts`) — added full context state reset at the top of every `renderEditor` call:
```
ctx.globalAlpha = 1
ctx.globalCompositeOperation = "source-over"
ctx.shadowColor = "transparent"
ctx.shadowBlur = 0
ctx.shadowOffsetX = 0
ctx.shadowOffsetY = 0
ctx.imageSmoothingEnabled = true
ctx.filter = "none"
```
Guarantees a clean slate every frame regardless of what previous draw functions left behind.

**Scroll event batching** (`CanvasEditor.tsx`) — replaced per-event `setScrollTop()` with a delta accumulator:
- Wheel events add `e.deltaY` to `pendingScrollDelta`
- A single `requestAnimationFrame(flushScroll)` drains the accumulator once per frame
- One signal write → one effect trigger → one render per frame, no matter how many wheel events fired

### Also Applied
- `{ alpha: false }` on `getContext("2d")` for all opaque canvases (editor, terminal, git graph, git page, welcome, tour) — minor perf improvement, prevents compositor alpha blending

### Glyph Atlas Performance Regression — Diagnosed

The recent "Tier 1 optimizations" (glyph atlas, offscreen renderer, line cache) made file opening noticeably slower and degraded overall performance.

**Root causes identified:**

1. **Glyph atlas adds overhead, not savings.** Every visible character now goes through `editorAtlas.drawText()` which does a `Map.get()` lookup (with string key concatenation per char) + `ctx.drawImage()` from an OffscreenCanvas. On cold cache (first render of any file), it does `fillText()` on the atlas *plus* `drawImage()` — strictly more work than direct `fillText()`. Modern browsers (Chrome/WebKit in Tauri) already have an internal Skia glyph cache that does this at a lower level with zero JS overhead. The JavaScript-level atlas is redundant.

2. **Offscreen renderer + render worker are dead code.** `createOffscreenRenderer` is never imported or called anywhere. `CanvasEditor.tsx` calls `renderEditor()` directly on the main thread. The worker files (`offscreen-renderer.ts`, `render-worker.ts`) ship in the bundle (13 KB) but do nothing.

3. **`fileIsLarge()` IPC round-trip on every file open.** Even tiny files hit the Rust backend to check `fs::metadata().len()` before loading. Adds unnecessary latency to every file open.

**Fix applied — 9 issues resolved:**

1. **canvas-renderer.ts** — Replaced 22 `editorAtlas.drawText/drawChar` call sites with direct `ctx.fillText()` via `monoText()` helper. Eliminates per-character Map lookups, string key concatenation, and OffscreenCanvas drawImage overhead.
2. **CanvasTerminal.tsx** — Killed permanent rAF loop. Now uses on-demand `scheduleTermRender()` that only fires when `needsRedraw` is set. Replaced `terminalAtlas.drawChar` per-cell with direct `fillText`. Canvas backing store only resizes when dimensions actually change.
3. **WelcomeCanvas.tsx** — Removed atlas usage for rain drops and particles. Fixed canvas resize every frame. Removed alpha-quantization hack.
4. **BusterProvider.tsx** — Removed `fileIsLarge()` IPC round-trip on every file open. Now tries `readFile` directly (fast path), falls back to chunked loading only if it fails.
5. **text-measure.ts** — Removed atlas cache clearing.
6. **Deleted 3 dead files** — `glyph-atlas.ts` (181 lines), `offscreen-renderer.ts` (58 lines), `render-worker.ts` (26 lines) = 265 lines removed.

**Result:** JS bundle 360 KB → 356 KB gzip. 97 tests pass. Noticeably faster file opens and smoother rendering.

## Session 5 — Image Viewer (Full Implementation)

### Problem
Clicking any image file (.png, .jpg, .gif, etc.) in the sidebar opened it as garbled binary text in CanvasEditor — typically showing a single unreadable character. The entire file pipeline assumed UTF-8 text: `fs::read_to_string()` in Rust, JSON serialization over IPC, and rendering through the text editor.

### Solution — New Image Tab Type + Dual-Path Loading

**Architecture:** Image files bypass the text pipeline entirely. They get their own tab type (`"image"`), their own lightweight open flow (no content loading, no LSP, no file watcher), and their own viewer component.

**Dual-path image loading:**
1. **Primary:** Tauri's `convertFileSrc()` asset protocol — zero overhead, browser decodes natively
2. **Fallback:** New `read_binary_file` Rust command reads raw bytes, detects MIME from magic bytes, returns base64 data URL over IPC

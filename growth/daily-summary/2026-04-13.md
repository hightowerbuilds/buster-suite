- **Settings UI** (`SettingsPanel.tsx`): Button selector under "Blog Mode Theme" — Normal / Newspaper. Uses shared `BLOG_THEMES` constant from the resolver module.

### Newspaper Theme (NYT-Inspired)

A full visual theme for markdown blog mode previews, inspired by The New York Times:

- **Typography**: Georgia/Times serif body, UnifrakturMaguntia (Old English) for H1 masthead
- **H1**: 2.8em with 3px double-rule underline, Old English font
- **H2**: 1.8em with single-rule underline
- **H3**: Italic serif
- **H4**: Uppercase monospace (Courier), small caps feel
- **Drop caps**: 3.2em floated first-letter after H1/H2
- **Body text**: Justified with auto-hyphenation
- **Blockquotes**: Editorial pull-quote style (top border, italic, no left border)
- **Tables**: Uppercase Courier headers, horizontal rules only (no cell borders)
- **Strong**: Italic instead of color (newspaper convention)
- **HRs**: Narrow centered divider (40% width)
- **Code blocks**: Square corners, thin border, monospace inset

### Dark/Light Mode Integration

Both newspaper variants use `palette.editorBg` for background, matching the app's actual theme:
- **Dark mode**: `#1e1e2e` (dark blue-black) with warm light text (`#e0ddd5`), muted rules (`#3a3835`)
- **Light mode**: `#e8e0d4` (oatmeal) with dark ink (`#121212`), silver rules (`#ccc`)

### Token Resolver Architecture (`blog-themes.ts`)

Replaced the fragile CSS class-toggling approach (`.blog-theme-newspaper`, `.blog-theme-newspaper-dark`) with a single-source-of-truth token resolver.

**The problem**: CSS variant classes caused specificity wars, leftover variables from removed classes, and ordering bugs when `theme_mode` and `blog_theme` changed simultaneously.

**The solution**: `resolveBlogTokens(blogTheme, themeMode, palette) → Record<string, string>`

- **`BlogTokens` interface**: ~60 named tokens covering every visual attribute — typography, colors, heading decoration, paragraph alignment, drop caps, link style, blockquote borders, code styling, table formatting, image borders, spacing
- **Pure functions per variant**: `normalTokens(palette)`, `newspaperLightTokens(palette)`, `newspaperDarkTokens(palette)` — each returns a complete `BlogTokens` object
- **`tokensToVars()`**: Maps tokens to `--blog-*` CSS custom properties
- **`BLOG_THEMES` constant**: Shared between resolver and settings UI — single place to add themes

**BlogPreview component**: One `createMemo` computes all tokens reactively from `(props.theme, store.settings.theme_mode, store.palette)`. Single `blog-preview` CSS class always — no class toggling.

**CSS stylesheet**: Purely structural. Every color, font, spacing, and border value is a `var(--blog-*)` reference. Zero hardcoded visual values, zero theme-variant selectors. Adding a new theme touches zero CSS.

**Scaling**: To add a new theme, add one token function + one entry in `BLOG_THEMES`. To add a new token, add one field to the interface, one value per theme function, one line in `tokensToVars()`, and one `var()` in CSS.
